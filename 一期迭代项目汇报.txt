================================================================================
               SeekFlow 短视频平台 — 一期迭代结项汇报稿
================================================================================

各位好，今天我代表项目组向大家汇报 SeekFlow 短视频平台一期迭代的交付情况。

首先解释一下我们的项目名称——SeekFlow。Seek 代表探索与发现，Flow 代表
信息流与心流体验。我们希望用户打开 SeekFlow 的那一刻，就能沉浸到内容发
现的愉悦状态中去。这个名字也承载了我们的技术理念——让数据像水流一样在系
统各组件之间高效、可靠地流动。

本次汇报分为五个部分。第一部分简要回顾一期的建设目标，第二部分深入讲解
我们的技术架构设计，第三部分逐个子系统展开交付细节，第四部分给出量化指标，
最后预告二三期的演进方向。


【第一部分：一期目标】

SeekFlow 的整体建设规划是三个迭代周期。一期作为整个项目的基石，我们的
核心命题只有一个——打通"内容消费—身份认证—社交互动"这条完整的业务闭环。

具体来说，一期交付了三个子系统。第一，视频内容消费系统，用户能刷到推荐
视频，能点赞、评论、收藏、分享，也能自己拍摄上传。第二，统一身份认证体
系，支持邮箱密码登录和验证码登录两种方式，基于 JWT 实现无状态会话管理。
第三，实时消息与通知中心，用户之间可以实时私聊，系统能推送点赞、评论、
关注等多类型的通知。

一句话概括一期的成果：一个用户从注册登录，到刷视频互动，再到给朋友发私
聊消息，这条完整的体验路径已经全部跑通了。


【第二部分：技术架构】

接下来我想重点讲讲我们在一期搭建的技术架构。这是一套面向生产环境设计的
分布式系统，虽然目前是一个单体部署，但架构分层上已经为未来的微服务拆分
预留了清晰的边界。

先看整体拓扑。最上层是客户端层，我们选择了 Vue 3.5 配合 TypeScript 和
Vite 6 构建工具。为什么选择这套组合？Vue 3 的响应式系统基于 Proxy 重构，
性能和内存占用相比 Vue 2 有了质的提升，而且它的 Composition API 天然与
TypeScript 的类型推断深度整合。Vite 6 作为下一代构建工具，开发服务器冷
启动在五百毫秒以内，生产构建基于 Rollup 做代码分割，产物体积控制得相当好。

状态管理我们统一使用 Pinia，这是 Vue 官方推荐的新一代方案。它不像 Vuex
那样依赖字符串魔法常量，每一个 Store 都有完整的 TypeScript 类型推导，
而且天然支持按路由懒加载，对首屏性能非常友好。

往下走是网关接入层，Spring Boot 3.3 作为我们的微服务基座。这里有几个关
键选型值得展开。第一，我们直接上了 Java 17，这意味着可以享受 Virtual
Threads 带来的吞吐量红利，后续做高并发优化时不需要改代码架构。第二，ORM
层选择 MyBatis-Plus 3.5.7，它的 Lambda QueryWrapper 让我们在编写复杂查
询时依然有编译期的字段校验，避免了传统 MyBatis XML 字符串拼写错误的隐患。
内置的逻辑删除、自动填充、分页插件也大量减少了样板代码。

再往下看数据层，这里是我们投入设计精力最多的部分。数据库采用 MySQL 8.0
配合 InnoDB 事务引擎，一期建了八张业务表，覆盖用户、视频、评论、点赞、
关注、私聊消息和通知这几个核心域。文件存储则引入了 MinIO——一个 S3 兼容
的对象存储，我们按视频和图片分了两个隔离的 Bucket。为什么要额外引入对象
存储而不是直接放服务器本地磁盘？一方面是存储容量的可扩展性，另一方面是
MinIO 支持预签名 URL，前端可以直接拿到临时上传地址，视频文件不需要经过
后端服务器中转，大幅降低了带宽压力。

到这里我想重点介绍两个为高并发场景准备的基础设施组件。

第一个是 Apache Kafka 消息队列。为什么一个短视频应用需要消息队列？核心
场景是私聊消息的异步削峰。想象一下，如果某个直播间引爆了流量，瞬间有几
万条私聊消息涌入，如果每条消息都同步写数据库，MySQL 的连接池瞬间就会被
打满。我们的方案是——消息先发到 Kafka，由 Consumer 批量消费后写入 MySQL。
Kafka 的 Producer 端我们配置了 Snappy 压缩和十六 KB 的批量发送策略，
Consumer 端配置了手动确认加每次拉取两百条的批量处理。而且我们还做了一
个实用设计——Kafka 可以通过配置开关一键降级为数据库直写模式，开发环境或
者 Kafka 集群维护时不需要改一行代码。

第二个是 WebSocket 实时通信。我们基于 Spring 原生的 WebSocketHandler
实现了一套长连接管理方案。核心组件是两个——ChatWebSocketHandler 负责处
理握手鉴权和消息路由，SessionManager 用一个 ConcurrentHashMap 维护
userId 到 Session 的映射关系，支持同一个用户多设备同时在线。连接建立时
通过握手拦截器校验 JWT 令牌，连接断开时自动清理会话映射。前端方面，我们
用不到七十行代码封装了一个 WebSocket 管理器，实现了消息类型自动分发和断
线三秒自动重连，对上层业务代码完全透明。

关于安全性，我们在一期建立了四层纵深防御。传输层走 HTTPS 加密信道，认证
层用 JWT 的 HMAC-SHA256 签名算法保证令牌不可伪造，存储层用 BCrypt 十分
钟级别的计算成本做密码哈希，应用层则在 Axios 响应拦截器中统一捕获 401
状态码并引导用户重新登录。


【第三部分：子系统交付详述】

现在让我逐个子系统展开交付的具体内容。

首先是视频内容消费系统，一期的完成度大约是百分之八十五。这个子系统承载
了 SeekFlow 最核心的用户体验。

推荐信息流这块，我们实现了基于分页游标的无限滚动加载方案。跟传统的
offset 分页不同，游标分页在数据频繁插入的场景下不会出现重复或遗漏，这
对于视频推荐这种实时更新的信息流来说至关重要。视频播放策略上，我们做了
视口交叉检测——视频进入可视区域时自动开始播放，滑出可视区域时自动暂停并
回收播放器资源。还有一个很细节但用户体验感知很强的地方——全局静音开关和
单独的视频音量控制，用户可以在首页一键切换静音状态，这个状态通过事件总
线广播到所有视频播放器组件。

搜索引擎支持视频标题、音乐名称、用户昵称三个维度的检索，另外还有热搜榜
单和搜索历史记录。技术上没有引入 Elasticsearch，而是用 MySQL 的 LIKE
模糊匹配加上合理的索引设计来支撑，考虑到一期用户量和数据量，这个方案性
能完全够用，也避免了过早引入复杂中间件。

社交互动矩阵涵盖了四个核心动作——点赞、评论、收藏、分享。点赞这块我们采
用了乐观更新策略，用户点击爱心图标后 UI 立刻响应，不等待后端返回，如果
后端请求失败再回滚状态。这个设计让交互反馈几乎是零延迟的，跟抖音原生的
体验非常接近。评论系统支持一级评论和多级嵌套回复，评论本身也可以被点赞，
数据结构上用 parent_id 字段构建了一棵评论树。分享功能我们做了一个比较
完整的分享弹窗，支持站内转发、复制链接等渠道，每次分享后端都会记录计数。

UGC 内容发布是另一个重头戏。我们调用了浏览器的 MediaRecorder API 来获
取摄像头画面，支持拍照、六十秒、三分钟和十分钟四种录制模式，可以切换前
后摄像头和控制闪光灯。录制完成后进入三段式流程——先预览视频，然后选取封
面帧，最后编辑描述和音乐信息后上传。视频文件通过 MinIO 的预签名 URL 直
接上传到对象存储，后端只负责接收元数据并在视频表写入记录，文件本身不走
服务器中转。

这个子系统对应的后端是 VideoController，一个三百四十五行、十四个接口的
控制器，覆盖了推荐流、历史记录、作品管理、点赞收藏分享、评论体系等全部
功能。配合它的 VideoService 有四百多行的业务逻辑，CommentService 有八
百五十多行处理评论树的增删改查。

再看第二个子系统——统一身份认证体系，完成度约百分之九十。

我们实现了双通道登录策略。通道 A 是经典的邮箱加密码方案，密码用 BCrypt
做不可逆哈希后存入数据库，登录时做哈希比对，校验通过后签发七天有效的
JWT 令牌。通道 B 是邮箱验证码方案，通过 QQ 邮箱的 SMTP 加密通道发送六
位数字验证码，用户输入验证码后核验通过直接签发 JWT。我们还预留了微信、
QQ、微博和苹果四种第三方授权登录的 UI 入口，后续接上 OAuth 协议即可激活。

JWT 令牌的管理我们设计了一个完整的生命周期——从登录签发开始，令牌存到
localStorage，Axios 请求拦截器自动往每个请求的 Authorization 头附加
Bearer 令牌，Pinia Store 同时维护全局的用户信息状态，包括昵称、头像、
地区、学校等画像数据。退出登录时清空令牌、销毁用户状态、断开 WebSocket
连接，一气呵成。

还有一个关于注册流程的考量。为什么我们同时做了邮箱验证码注册和邮箱密码
注册两种方式？这是为了降低不同用户群体的注册门槛——年轻用户习惯快捷的验
证码方式，而注重安全的用户可以选择传统的密码方式。两种方案共用同一套
BCrypt 加密和 JWT 签发基础设施。

第三个子系统——实时消息与通知中心，完成度约百分之八十，也是技术复杂度最
高的一个子系统。

私聊系统的核心设计理念是"先落库，再推送"，这是保证消息零丢失的关键原则。

我描述一下一条消息的完整旅程。用户在聊天界面输入消息点击发送，前端调用
REST 接口将消息体提交到 MessageController。Controller 不做复杂处理，直
接将消息交给 Kafka Producer 投递到名为 seekflow-chat-message 的 Topic。
Kafka Consumer 端批量拉取消息，经过 Snappy 解压后逐条写入 MySQL 的
message 表。落库成功后，Consumer 检查接收方是否在线——如果在线，通过
SessionManager 拿到接收方的 WebSocket Session，实时推送这条消息；如果
不在线，消息已经安全地存在数据库里了，接收方下次打开聊天页面时会通过历
史消息接口拉取到。

这个架构有几个精妙之处。第一，Kafka 充当了削峰填谷的缓冲层，不管瞬时消
息量多大，数据库承受的写入速率都是由 Consumer 的批量处理能力决定的，不
会被瞬间打爆。第二，消息发送和消息推送是完全解耦的，发送接口的响应时间
只取决于写入 Kafka 的速度，通常是毫秒级。第三，我们用配置开关
douyin.kafka.enabled 控制是否走 Kafka 链路，关掉后消息直接同步写库，开
发环境不必安装 Kafka 也能完整调试聊天功能。

通知中心聚合了五种类型的系统通知——互动通知包括点赞、评论和关注，系统通
知来自抖音助手，还有任务通知、直播通知和钱包通知。每种通知走独立的未读
计数，前端按分类 Tab 展示。通知的触发方式也很精巧——比如用户 A 点赞了用
户 B 的视频，VideoController 在处理点赞逻辑的同时，会通过 Kafka 异步发
送一条通知事件，由 Consumer 消费后写入 notification 表，如果 B 在线就
WebSocket 实时推送。

会话列表的功能逻辑是这样的——将每个用户的最近对话按最后一条消息的时间倒
序排列，每条会话展示对方的头像、昵称、最后一条消息摘要和未读红点数字。
用户进入某个聊天页面时，系统自动触发 markRead 接口将该会话的所有未读消
息标记为已读。

另外我们还实现了一系列扩展功能——粉丝列表和访客记录的社交入口、已加入的
群聊列表、红包领取详情、好友备注名设置、以及从视频或商品详情页直接分享
转发到聊天的功能。


【第四部分：量化数据】

汇报完三个子系统的细节，我们来看一组数字，直观感受一期的交付规模。

前端方面，我们累计开发了九十二个 Vue 页面组件和三十七个通用组件，定义了
超过五十条前端路由，用到了 Vue Router 的懒加载、过渡动画和 KeepAlive
缓存等高级特性。

后端方面，总共六十七个 Java 类文件，暴露了四十二个 RESTful API 端点，
配合八张数据库表构成了完整的数据模型。

前后端代码总规模约一万五千行。通信协议覆盖了 HTTP RESTful、WebSocket
长连接和 Kafka 异步消息队列三条链路。

这些数字背后反映的是一个事实——SeekFlow 一期已经不是简单的 Demo 或者课
程作业了，而是一个具备生产级架构思想的中型全栈项目。


【第五部分：后续规划】

最后我简要汇报一下二三期的规划方向。

二期将横向扩展业务版图，目前规划了三大板块。商城系统排在第一位，这也是
五个核心板块中唯一一期尚未触及的，需要从商品列表浏览、购物车管理到订单
创建和模拟支付完成一条完整的电商链路。个人中心的增强主要是补齐作品和收
藏的管理能力，以及更丰富的个人主页展示。社交发现模块包括找熟人、用户主
页详情和关注粉丝列表。

三期将聚焦非功能性需求，目标是让 SeekFlow 达到可以真正上线运行的标准。
性能方面，前端要做首屏关键渲染路径优化和视频预加载策略，后端要引入
Redis 做热点数据缓存、优化慢查询索引。质量保障方面要补齐单元测试和端到
端测试。DevOps 方面要搭建 CI/CD 自动构建部署流水线，实现 Docker 容器化
部署。可观测性方面要接入应用指标监控和集中日志系统。

以上就是 SeekFlow 一期迭代的完整汇报。我们从零开始，在一期搭建了一个涵
盖视频内容消费、身份认证和实时消息三大核心能力的全栈短视频平台，技术选
型兼顾了当下的开发效率和未来的扩展空间。感谢各位的聆听，有什么问题欢迎
交流。
